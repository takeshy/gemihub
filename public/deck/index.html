03設計思想 — 個人情報ゼロ預かり・ロックインなし

「預からない」が、
最大のセキュリティ対策。

データ実体はすべて ユーザー自身のGoogle Drive に置く。GemiHubはDBを持たない。
サービスが消えても、ユーザーのファイルはそのまま残る。それがロックインなしという意味。
DB
DBを持たない
データ実体はすべて ユーザーのGoogle Drive上。サービス側に蓄積しない。
// no-backend-db
◴
スコープは drive.files のみ
アプリが作ったファイル以外は見えない / 触れない。ユーザーの他のDrive資産に一切干渉しない。
// OAuth: drive.files
⎋
サービスが消えても残る
ファイルは Google Driveの.mdや.json。GemiHubが閉じても、あなたのデータはあなたの手元に。
// lock-in: none
🔒
RSA + AES ハイブリッド暗号化
秘匿データは AIにすら中身を見せない。必要な場面だけユーザーがパスワードで復号する。
// E2E encrypted at rest
⇄
IndexedDB + Service Worker
オフラインで編集 → Push / Pull で同期。ネットがなくても書き続けられる。
// offline-first

05 / 07
06まとめ / Closing

集約して、暗号化して、ビジュアルで組む。

1
AIが賢くなるほど、データ自体の価値が高まる。
秘匿情報も含めて一箇所に集約することで、管理を楽にしたい。
2
でも秘匿情報をAIに見られるのは抵抗がある。
暗号化で守り、必要な場面だけユーザーがパスワードで復号する。
3
処理をビジュアルで見えれば、非エンジニアもバイブコーディングできる。
プログラマでさえコードより挙動でレビューする時代。
4
Plan / Generate / Review は高品質モデル、実行は安いモデル。
SkillとWorkflowの設計思想が、そのままコスト最適化になる。
5
Proプランでは、Webサービスごと構築できる。
カスタムドメインでAIがWebページとAPIを実装 (例: 面接予約ポータル)。
